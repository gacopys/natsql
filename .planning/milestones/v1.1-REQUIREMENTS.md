# Requirements: natsql — v1.1 Tech Debt Cleanup

**Defined:** 2026-05-29
**Core Value:** A developer can define a materialized view from a stream, publish events, and query the current state with `SELECT ... WHERE ...` — zero infrastructure beyond NATS.

## v1.1 Requirements

### Query Engine

- [ ] **FIX-ENG-01**: PK equality query applies non-PK WHERE conditions as post-filter (was silently dropping them)
- [ ] **FIX-ENG-02**: Engine.Query is safe for concurrent access (data race on Engine.kv eliminated)
- [ ] **FIX-ENG-03**: filterRow uses type-aware comparison instead of fmt.Sprint type coercion
- [ ] **FIX-ENG-04**: SQL parser accepts boolean literals (true/false) in WHERE clause values

### Materializer

- [ ] **FIX-MAT-01**: PK values are sanitized against KV key special characters (/, *, >, .)
- [ ] **FIX-MAT-02**: DLQ publish failures are surfaced; events are not acknowledged on DLQ failure
- [ ] **FIX-MAT-03**: Engine.Start cleans up correctly on partial initialization failure
- [ ] **FIX-MAT-04**: JSON integer values >2^53 preserve full precision (avoid float64 truncation)

### Transport & Hardening

- [ ] **FIX-TRN-01**: HTTP server has read, write, and idle timeouts configured
- [ ] **FIX-TRN-02**: HTTP query endpoint enforces a maximum request body size
- [ ] **FIX-TRN-03**: NATS query handler uses a bounded context with timeout
- [ ] **FIX-TRN-04**: Dead code, unused parameters, and test flakiness sources cleaned up

## Future Requirements (Deferred)

### Query Engine v2

- **QUERY-02**: User can query KV state with `SELECT ... WHERE <col> > <val> / <col> < <val>` (range scan)
- **QUERY-03**: `LIMIT` support on query results
- **INDEX-01**: Secondary indexes on materialized views (single-column equality + range)

## Out of Scope

| Feature | Reason |
|---------|--------|
| New feature development | v1.1 is exclusively tech debt cleanup; new features deferred to v2.0.0+ |
| Full pgwire / PostgreSQL protocol | Massive surface area; v1 needs a stable query engine first |
| JOINs across views | Query planning complexity; needs multi-KV-bucket coordination |
| Aggregations (COUNT, SUM, GROUP BY) | Needs state tracking across events |
| Push queries / subscriptions | Adds server-side complexity for maintaining query results over time |
| Schema migration / ALTER VIEW | Requires full re-materialize — document as workaround |
