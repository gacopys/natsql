# Requirements: natsql — v1.0 (Archived)

**Defined:** 2026-05-22 | **Shipped:** 2026-05-28
**Status:** All v1 requirements complete

## v1 Requirements

### Materialization

- [x] **MAT-01**: User can define a materialized view in a YAML config file (source stream subject, key column, column mapping)
- [x] **MAT-02**: Materializer consumes JetStream events from the configured stream with a durable consumer (survives restarts)
- [x] **MAT-03**: Materializer maintains a KV bucket with current row state (PK → JSON value)
- [x] **MAT-04**: Materializer handles malformed events gracefully (log + skip, continue processing)

### Query

- [x] **QRY-01**: User can query by primary key: `SELECT * FROM view WHERE <pk_col> = <val>`
- [x] **QRY-02**: Query engine validates column names exist in the view schema
- [x] **QRY-03**: Query engine returns typed JSON results

### Interface

- [x] **IFC-01**: Query via NATS request-reply (`nc.Request("natsql.query", sqlBytes)` → JSON response)
- [x] **IFC-02**: Query via HTTP/JSON API endpoint (`POST /api/v1/query`)
- [x] **IFC-03**: Query via Go library API (`engine.Query(ctx, "SELECT ...")`)

### Deployment

- [x] **DEP-01**: Standalone `natsql server` binary with embedded NATS (single-node, dev mode)
- [x] **DEP-02**: Standalone `natsql server` binary connects to external NATS cluster
- [x] **DEP-03**: Importable as Go library (`natsql.New(js, config)`)
- [x] **DEP-04**: Graceful shutdown — drain materializer consumer, flush pending writes, stop HTTP server
