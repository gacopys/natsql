    **Part A: Consumer drain in materialize.Run()**

    Modify `materialize.Run()` to accept a `drainCh <-chan struct{}` parameter. When drainCh is signaled, the bridge goroutine calls `cons.Drain()` before exiting, which prevents unnecessary redeliveries on restart.

    New signature:
    ```go
    func Run(ctx context.Context, js jetstream.JetStream, viewCfg *natsql.ViewConfig, bucket jetstream.KeyValue, dlqStream jetstream.Stream, logger *slog.Logger, drainCh <-chan struct{}) error
    ```

    In the bridge goroutine, add drainCh handling. The challenge is that `msgCtx.Next(NextContext(ctx))` blocks until a message arrives or ctx is done. Use a child context pattern:

    1. Create `fetchCtx, fetchCancel := context.WithCancel(ctx)` before the bridge goroutine
    2. Start a monitor goroutine that selects on drainCh and ctx.Done():
       - If drainCh received: call `fetchCancel()` (unblocks NextContext), then call `cons.Drain()`
       - If ctx.Done(): call `fetchCancel()` and return
    3. The bridge goroutine's Next uses `NextContext(fetchCtx)`
    4. When NextContext returns context.Canceled and the main ctx is still alive, it's a drain signal — exit normally

    ```go
    // Bridge goroutine with drain support
    fetchCtx, fetchCancel := context.WithCancel(ctx)
    go func() {
        defer close(msgCh)
        defer fetchCancel()
        for {
            msg, nextErr := msgCtx.Next(jetstream.NextContext(fetchCtx))
            if nextErr != nil {
                if errors.Is(nextErr, context.Canceled) {
                    // Either main ctx cancelled or drain requested
                    // Check if drain was the cause
                    select {
                    case <-drainCh:
                        // Drain requested: call cons.Drain() before exit
                        if drainErr := cons.Drain(); drainErr != nil {
                            logger.Warn("consumer drain error", "view", viewCfg.Name, "error", drainErr)
                        }
                    default:
                        // Main context cancelled, just exit
                    }
                    return
                }
                if errors.Is(nextErr, jetstream.ErrMsgIteratorClosed) {
                    return
                }
                logger.Error("consumer Next error", "view", viewCfg.Name, "error", nextErr)
                time.Sleep(100 * time.Millisecond)
                continue
            }
            select {
            case msgCh <- msg:
            case <-fetchCtx.Done():
                return
            }
        }
    }()

    // Monitor drain signal separately
    go func() {
        select {
        case <-drainCh:
            fetchCancel()
        case <-ctx.Done():
            fetchCancel()
        }
    }()
    ```

    **Part B: Engine Close() reordering per D-57**

    Replace the current `Close()` method:

    ```go
    func (e *Engine) Close() error {
        e.mu.Lock()
        defer e.mu.Unlock()

        if !e.started {
            return ErrNotStarted
        }

        // D-57 Step 1: Stop HTTP server (5s timeout for in-flight requests)
        if e.httpServer != nil {
            shutdownCtx, shutdownCancel := context.WithTimeout(context.Background(), 5*time.Second)
            if err := e.httpServer.Shutdown(shutdownCtx); err != nil {
                e.logger.Warn("HTTP server shutdown error", "error", err)
            }
            shutdownCancel()
            e.httpServer = nil
        }

        // D-57 Step 2: Unsubscribe NATS query handler (prevents new query requests)
        if e.natsSub != nil {
            if err := e.natsSub.Unsubscribe(); err != nil {
                e.logger.Warn("NATS unsubscribe error", "error", err)
            }
            e.natsSub = nil
        }

        // D-57 Step 3: Signal drain to all materializer consumers
        // This triggers cons.Drain() in each materializer before context cancel
        for _, ch := range e.drainChans {
            close(ch)
        }
        // Give drain a moment to propagate (drain is async but fast)
        e.drainChans = nil

        // D-57 Step 4: Cancel materializer context
        if e.cancel != nil {
            e.cancel()
        }

        // D-57 Step 5: Wait for all goroutines (materializers + HTTP monitor)
        e.wg.Wait()

        // D-57 Step 6: Close embedded NATS — handled by facade (natsql package)
        // The engine package does not own embedded NATS lifecycle

        e.started = false
        e.kv = nil
        e.logger.Info("engine closed")
        return nil
    }
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Embedded NATS ↔ Engine | Both run in the same process; no network boundary between embed server and engine |
| Engine → JetStream KV | KV operations are in-process when using embedded NATS, or network calls when external |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-01 | Denial of Service | Engine.Close() shutdown hang | mitigate | HTTP server shutdown has 5s timeout. Consumer drain is async (cons.Drain() returns immediately). WaitGroup.Wait() has no timeout but all goroutines respond to context cancel. |
| T-03-02 | Elevation of Privilege | Embedded NATS access | accept | Embedded NATS is in-process only, binds to 127.0.0.1. No external network access. Same security boundary as host process. |
